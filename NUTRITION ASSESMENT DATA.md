# Nutrition Assessment Data — Legacy Google Sheet

> **Source:** `NUTRITION ASSESMENT DATA.pdf` — This is the **actual exported data** from the current Google Sheets. It represents all historical patient assessments that must be migrated into NAMS.
>
> **How the outlet system currently works:** There is **no single unified Google Sheet**. Instead, there are **5–6 separate Google Forms** — one per outlet — each saving to its own dedicated Google Sheet. The PDF shown here is a combined export. The outlet a row belongs to is determined entirely by **which form/sheet it was exported from** — there is no outlet column in the data itself. This is a fundamental limitation of the current Google Forms setup that NAMS is designed to fix.
>
> **Migration process:** Each outlet's sheet must be exported as a **separate CSV** and passed to the migration script with the outlet name as a parameter (e.g., `npx ts-node scripts/migrate.ts --outlet="Makkaraparamba"`). The script then tags every row from that CSV with the correct outlet.
>
> **Outlets are increasing:** New outlets are added regularly. As of the data in this PDF, the known outlets are: **Makkaraparamba** (mkp), **Manjeri**, **Karikkallathani** (kkl), **Tirur**, **Melattur**, **Perinthalmanna** (pmna), **Pariyappuram**. More will follow — the Admin panel in NAMS handles this dynamically without code changes.

## Column Structure (18 Columns)

| # | Column Name | Type | Notes |
|---|---|---|---|
| 1 | `Timestamp` | DateTime | Auto-generated by Google Forms on submission. **Discard** — not needed in NAMS. |
| 2 | `DATE` | Date string | The actual assessment date entered by the nutritionist. Often differs from Timestamp. Use this as `Assessment.date`. |
| 3 | `NAME` | String | Patient full name. Mixed case, trailing spaces common. |
| 4 | `AGE` | String / Int | Usually a number. Sometimes blank or `.` — store as `null`. |
| 5 | `SEX` | String | `Male`, `Female`, `female`, `f`, `fe`, `m` — normalize to `MALE`/`FEMALE`/`OTHER`. |
| 6 | `OCCUPATION` | String | Often `.`, `nil`, `not known`, blank — store as `null`. |
| 7 | `PLACE` | String | Patient's home area (not the outlet). Often shortened (e.g., `mkp`, `mjri`, `mnjri`, `kariklthni`). |
| 8 | `HIGHT` | String / Float | Height in cm. Frequently `not known`, blank, or `.` — store as `0`. Note: column name has a typo (`HIGHT` not `HEIGHT`) — keep as-is to match sheet. |
| 9 | `WEIGHT` | String / Float | Weight in kg. Same handling as HIGHT. |
| 10 | `BMI` | String / Float | Pre-entered BMI. Often blank, `not known`, or a number. NAMS recalculates from height/weight — use this only as a fallback check. |
| 11 | `CONTACT NUMBER` | String | 10-digit phone number — **Patient UID**. Strip spaces and dashes. Many entries in legacy data have wrong digit counts (11 digits, 12 digits, missing entirely) — store what's present and flag. |
| 12 | `NAMES OF TEST CONDUCTED` | String | Comma-separated test names. Mixed case, inconsistent spelling (e.g., `lipid`, `LIPID`, `lioid`, `lipd`). Split and store as JSON array. |
| 13 | `VARIATION IN RESULTS` | String | Free-text abnormal findings. Often `.`, blank, or `no` — store as `null`. |
| 14 | `DIET` | String | Dietary advice given. |
| 15 | `DO THEY NEED DIET PLAN` | String | `Yes`, `No`, `Maybe`, `NO`, `no` — normalize to consistent casing. |
| 16 | `REMARKS` | String | Call status or notes (e.g., `completed`, `not answering`, `not pick`, `call me later`). |
| 17 | `Result Received` | DateTime string | When lab results were received. Inconsistently formatted. Often blank in older rows. |
| 18 | `Patient Interaction` | DateTime string | When the nutritionist spoke to the patient. Inconsistently formatted. Often blank in older rows. |

> Some sheet tabs have a **Column 19** header — this appears to be a data artifact. Ignore it.

## Data Quality Issues (Critical for Migration)

These issues exist in the real data and the migration script must handle all of them:

| Issue | Examples | Resolution |
|---|---|---|
| **Missing contact number** | Several rows have no phone at all | Assign a generated placeholder UID; flag for manual review |
| **Invalid contact number length** | `94467881227` (11 digits), `62383228876` (11 digits), `558891100646` (12 digits) | Store as-is, flag as invalid — do not use as UID |
| **Shared contact number** | `9495090044` used by multiple different patients | Upsert by contact number — if name differs significantly, it may be a data entry error |
| **"not known" / "." / "nil" values** | Height, weight, occupation, place | Convert to `null` or `0` as appropriate |
| **Completely empty rows** | Some rows have only a name and no other data | Skip silently, log count |
| **Duplicate submissions** | Same patient, same date, same tests | Compare timestamp — skip exact duplicates |
| **Inconsistent date formats** | `26/02/2026`, `26/02/2026 12:29:34`, `1/2/2026`, `14/o2/2026` | Attempt to parse all; mark unparseable as `null` |
| **Typos in test names** | `lioid`, `lipd`, `cholestrol`, `tg,t.c`, `electyrolyte`, `pottasium` | Fuzzy match against master test list; flag unmatched tests |
| **"package" entries** | `1099 package`, `ramadan package`, `samvita`, `pckg` | These refer to clinic health packages — store as-is in `selectedTests`; Admin must map to formal test names |
| **Outlet not in column** | No `OUTLET` column exists | Must be inferred from the sheet tab name when exporting; legacy rows default to `"Legacy/Imported"` |
| **BMI already entered** | Some rows have a pre-filled BMI (e.g., `38.43`) | NAMS recalculates from height/weight; store original only if height/weight are missing |
| **Mixed SEX values** | `f`, `fe`, `Female`, `female`, `Male`, `m` | Normalize: anything starting with `f` → `FEMALE`, `m` → `MALE` |
| **"Result Received" / "Patient Interaction" absent** | Most rows from Nov 2025 – Jan 2026 have no timestamps | Store as `null`; these fields became required later |
| **Row `06/01/1900 shakkira 26 no does not exist`** | Literal text "does not exist" in the sheet | Skip — this was a manual note by the nutritionist |

## Outlet Migration Guide

Each outlet has its own Google Form and its own Google Sheet. Export each sheet separately as a CSV, then run the migration script once per outlet:

```bash
npx ts-node scripts/migrate.ts --outlet="Makkaraparamba"   --file="makkaraparamba.csv"
npx ts-node scripts/migrate.ts --outlet="Manjeri"          --file="manjeri.csv"
npx ts-node scripts/migrate.ts --outlet="Karikkallathani"  --file="karikkallathani.csv"
npx ts-node scripts/migrate.ts --outlet="Tirur"            --file="tirur.csv"
npx ts-node scripts/migrate.ts --outlet="Melattur"         --file="melattur.csv"
npx ts-node scripts/migrate.ts --outlet="Perinthalmanna"   --file="perinthalmanna.csv"
npx ts-node scripts/migrate.ts --outlet="Pariyappuram"     --file="pariyappuram.csv"
```

| Google Form / Sheet | Outlet Name in NAMS | Common abbreviations in PLACE field |
|---|---|---|
| Makkaraparamba form | Makkaraparamba | `mkp`, `pmna` (Perinthalmanna area) |
| Manjeri form | Manjeri | `manjeri`, `mjri`, `mnjri`, `majeri`, `m]jeri` |
| Karikkallathani form | Karikkallathani | `karikkallathani`, `kariklthni`, `kkl`, `karikktni` |
| Tirur form | Tirur | `tirur` |
| Melattur form | Melattur | `melattur`, `mellattur` |
| Perinthalmanna form | Perinthalmanna | `pmna`, `perinthalmanna` |
| Pariyappuram form | Pariyappuram | `pariyappuram` |

> **Important:** The `PLACE` column is the **patient's home area**, not the outlet. Do not use it to determine the outlet — use the CSV filename/source only.
>
> **New outlets:** As new outlets are added, Admin creates them in NAMS first via the Outlet Manager, then runs the migration script for that outlet's CSV. No code changes are needed.
